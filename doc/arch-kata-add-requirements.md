# アーキテクチャ・カタ研修アプリ 追加機能：要件確認ステップ

## 追加機能の概要

既存のワーク画面に**「要件確認」ステップを追加**する。
参加者がお題に対して「確認したい質問」と「想定される回答」をセットで自由入力し、
その要件認識をもとにアーキテクチャ特性・コンポーネント選択に進む流れにする。

本来のアーキテクチャ・カタで行う「顧客（モデレーター）への質問フェーズ」を
自己完結型で再現することで、「何を確認すべきか考える力」を鍛える。

---

## 既存画面への追加位置

```
Step 1：メンバー名入力              （既存）
Step 2：要件確認                   ★新規追加
Step 3：特性カード選択              （既存・番号繰り下げ）
Step 4：理由・トレードオフ入力      （既存・番号繰り下げ）
Step 5：コンポーネント選択          （既存・番号繰り下げ）
Step 6：確認・提出                 （既存・番号繰り下げ）
```

---

## DBへの追加カラム

`selections` テーブルに以下のカラムを追加するマイグレーションを作成してください。

```sql
ALTER TABLE selections
  ADD COLUMN requirements JSONB DEFAULT '[]'::jsonb;
```

### JSONBの構造

```json
[
  {
    "id": "1",
    "question": "同時アクセス数はどのくらいですか？",
    "answer": "ピーク時で1万人を想定しています"
  },
  {
    "id": "2",
    "question": "障害時の復旧目標時間（RTO）はありますか？",
    "answer": "法的に30分以内の復旧が求められています"
  }
]
```

## 新規Stepのコンポーネント（UI仕様）

`components/RequirementsEditor.tsx` を新規作成してください。

### 表示仕様

**ヘッダー部分**
- タイトル：「お題に対して確認したいことを整理しましょう」
- 説明文：「実際の開発では、設計を始める前に顧客へ要件を確認します。このお題に対して『何を確認すべきか』を考え、想定される回答も書いてみましょう。」

**入力エリア**
- 質問と想定回答のペアを入力するカード形式
- 最大5件まで追加可能
- 各カードの構成：
  ```
  ┌─────────────────────────────────┐
  │ 質問 #1                    [削除]│
  │ ┌─────────────────────────────┐ │
  │ │ 例）同時アクセス数はどのくら │ │
  │ │ いですか？                   │ │
  │ └─────────────────────────────┘ │
  │ 想定回答                         │
  │ ┌─────────────────────────────┐ │
  │ │ 例）ピーク時で1万人を想定し  │ │
  │ │ ています                     │ │
  │ └─────────────────────────────┘ │
  └─────────────────────────────────┘
  ```
- 「+ 質問を追加する」ボタン（5件未満のときのみ表示）

**参考ヒント（折りたたみ式）**
- 「確認すべき観点のヒント ▼」をクリックで展開
- 展開すると以下を表示：
  ```
  □ ユーザー数・アクセス規模はどのくらいか
  □ 障害時に許容できるダウンタイムはあるか
  □ データの機密性・セキュリティ要件はあるか
  □ 予算・開発チームの規模に制約はあるか
  □ 将来的なスケールアップの予定はあるか
  □ 法律・規制上の制約はあるか
  □ 既存システムとの連携は必要か
  ```

### バリデーション
- 質問欄が空のカードは提出時に警告（スキップは可能）
- 1件も入力せずに次へ進む場合は確認ダイアログを表示
  - 「要件確認をスキップしますか？スキップしても次のステップに進めます。」

## 発表用サマリー画面への追加表示

`/session/[joinCode]/group/[groupId]/summary` に以下を追加してください。

特性選択・コンポーネント選択の**前**に表示する。

```
## 確認した要件

Q1. 同時アクセス数はどのくらいですか？
A1. ピーク時で1万人を想定しています

Q2. 障害時の復旧目標時間（RTO）はありますか？
A2. 法的に30分以内の復旧が求められています
```

入力が0件の場合はこのセクションを表示しない。

## ファシリテーター管理画面への追加表示

各グループの回答確認部分に、要件確認の内容も表示してください。
特性・コンポーネントと同様にグループごとに確認できるようにしてください。

## 実装手順

### Step 1：DBマイグレーション
```
selectionsテーブルに requirements（JSONB型、デフォルト空配列）カラムを
追加するマイグレーションSQLを作成・実行してください。
```

### Step 2：RequirementsEditorコンポーネント作成
```
components/RequirementsEditor.tsx を作成してください。

Props：
  value: { id: string; question: string; answer: string }[]
  onChange: (requirements: { id: string; question: string; answer: string }[]) => void

機能：
- カード形式でQ&Aペアを入力できる
- 「+ 質問を追加する」で最大5件まで追加
- 各カードに削除ボタン
- 「確認すべき観点のヒント」折りたたみ表示
```

### Step 3：ワーク画面にStepを追加
```
/session/[joinCode]/group/[groupId] のステップ管理に
要件確認ステップ（Step 2）を追加してください。

既存のStep2（特性カード選択）の前に挿入し、
以降のStep番号を1つずつ繰り下げてください。

ステート管理に requirements を追加してください：
const [requirements, setRequirements] = useState<
  { id: string; question: string; answer: string }[]
>([]);
```

### Step 4：提出データにrequirementsを含める
```
selections テーブルへのINSERT処理に requirements（JSONB）を追加してください。
空配列の場合もそのままINSERTしてください。
```

### Step 5：サマリー画面に表示追加
```
サマリー画面でselectionsからrequirementsを取得し、
Q&A形式でカード表示の前に追加してください。
requirementsが空配列の場合はセクションごと非表示にしてください。
```

### Step 6：ファシリテーター画面に表示追加
```
ファシリテーター管理画面のグループ回答確認部分に
要件確認の内容（Q&A形式）を追加してください。
```

## 完成後のワーク画面フロー

```
Step 1：メンバー名入力
　↓
Step 2：要件確認 ★新規
　　- お題を読んで「確認したいこと」を質問形式で入力（最大5件）
　　- 各質問に「想定される回答」も入力
　　- ヒント一覧を参考にしながら考える
　↓
Step 3：アーキテクチャ特性カード選択（12枚→最大3つ）
　↓
Step 4：特性の理由・トレードオフ入力
　↓
Step 5：システムコンポーネント選択
　↓
Step 6：確認・提出
　↓
発表用サマリー表示
（要件確認 → 特性選択 → コンポーネント選択の順で表示）
```

## 教育上の意図（実装には不要・研修資料用メモ）

このステップを追加することで以下の学習効果が期待できる。

- 「何を確認すべきか」を考える力（要件定義の入口）
- 仮定を明示することで、特性選択の根拠が明確になる
- 発表時に「この要件を前提としたので、この特性を選びました」と説明できる
- グループ間で同じお題でも確認した要件が異なることが比較の議論を生む
